\## 1. 서론



\### 1.1 문서 목적 및 범위

본 프로젝트 계획서는 조직 내 협업을 위한 클라우드 파일 공유 시스템인 \*\*'미니 드라이브(Mini Drive)'\*\* 에 대한 전반적인 내용을 정리하여 프로젝트를 진행하는 데 있어 각 세부 사항의 지표로 삼는 것을 목적으로 한다.

본 문서는 다음과 같은 범위의 지표로 적용할 수 있다.

① 프로젝트 개요 (프로젝트 정의, 주요 기능 정리)

② 개발 계획 및 환경 (개발 절차 모형, 내용 및 일정, 소프트웨어/하드웨어 환경)

③ 팀 구성 (팀 구조 / 역할 및 책임)

④ 품질관리 (팀 미팅, 변경 사항 및 산출물 관리)

⑤ 산출물 (산출물 정의, 작성일 및 담당자)



\### 1.2 프로젝트 개요

\#### 1.2.1 프로젝트 정의

조직 내 구성원들이 원활하게 협업할 수 있도록 안전하고 빠른 데이터 업로드/다운로드를 지원하며, 체계적인 자료 공유가 가능한 클라우드 기반 파일 공유 시스템(미니 드라이브)을 구축한다.



\#### 1.2.2 주요 기능 설명

| ID | 설명 |
| :--- | :--- |
| F01 | 사용자 계정 생성 및 로그인 기능 |
| F02 | 클라우드 스토리지로의 파일 업로드, 다운로드 및 삭제 기능 |
| F03 | 조직원 간의 특정 폴더 및 파일 공유 기능 |
| F04 | 파일 목록 및 상태를 직관적으로 확인할 수 있는 UI 제공 |



\### 1.3 프로젝트 목표

사용자가 물리적인 저장 장치 없이 언제 어디서나 필요한 업무 파일에 접근할 수 있는 환경을 제공한다. 조직 내 협업 시 발생할 수 있는 파일 유실을 방지하고, 안정적인 클라우드 스토리지를 구축하는 것을 목표로 한다.



\### 1.4 용어 정의

| 용어 | 설명 |
| :--- | :--- |
| 클라우드 스토리지 | 인터넷을 통해 데이터를 저장, 관리, 유지보수, 백업할 수 있는 논리적인 스토리지 공간 |
| 동기화(Sync) | 클라이언트(PC/Mobile)의 폴더 상태와 서버의 저장소 상태를 동일하게 일치시키는 작업 |

\## 2. 개발 계획



\### 2.1 개발 절차 모형

\*\*1) 소프트웨어를 개발하기 위한 개발 절차 모형\*\*

사용자의 요구사항을 빠르게 반영하고 짧은 주기로 동작 가능한 소프트웨어를 확인하기 위해 Agile 방법론 중 \*\*SCRUM 모델\*\*을 사용하기로 결정하였다.



\*\*2) 개발 절차 모형과 함께 분석, 설계를 진행하기 위한 방법론\*\*

여러 가지 모델링 개념과 캡슐화를 지원하는 \*\*객체지향 방법론\*\*을 채택하였다. 클래스가 소프트웨어 모듈의 한 단위가 되어 분석, 테스트, 유지보수 등에 용이하다.



\### 2.2 WBS (Work Breakdown Structure)

\* \*\*요구분석:\*\* 계획서 검토 -> 기존 클라우드 시스템 조사 -> 요구 분석 및 정의 -> 필요한 기능 정의

\* \*\*설계:\*\* 분석서 검토 -> 시스템 구조 설계 -> 상세 설계 (UI 및 DB)

\* \*\*코딩:\*\* 서버 및 DB 구축 -> 기능의 구현 (업로드/다운로드 로직 등) -> 각 부분 기능의 통합

\* \*\*테스트:\*\* 전체적 테스트 -> 매뉴얼 작성 -> 발표 자료 작성



\### 2.3 개발 내용

| 개발 활동 | 설명 |
| :--- | :--- |
| 프로젝트 요약 | 프로젝트 요약서 작성 |
| 프로젝트 계획 | 프로젝트 계획서 작성 |
| 요구사항 정의/분석 | 요구사항 정의서 및 분석서 작성 |
| Product Backlog | 요구사항을 우선순위대로 순서화 |
| Sprint Planning Meeting | 스프린트가 시작될 때마다 백로그 선정 및 계획 |
| Daily Scrum Meeting | 매일 진행상황과 이슈 사항, 계획 공유 |
| Sprint | 짧은 주기로 프로그램을 개발하면서 피드백 제공 |
| Sprint Retrospective | 팀 내부에서 발생한 일들을 회고하며 개선할 점 도출 |
| Sprint Review | 스프린트 종료 시점에 각자 한 일 리뷰 및 문서화 |



\### 2.4 개발 일정

(프로젝트 기간에 제한을 두지 않으므로 3개월 단위의 스프린트 일정으로 구성)

\* \*\*1개월 차:\*\* 요구사항 정의 및 분석, 시스템 구조 및 DB 설계, UI 프로토타이핑

\* \*\*2개월 차:\*\* 핵심 기능(Sprint 1\~3: 회원가입, 파일 업로드/다운로드, 공유 기능) 개발

\* \*\*3개월 차:\*\* 기능 통합, 전체 테스트, 디버깅 및 최종 산출물 정리

\## 4. 품질 관리



\### 4.1 팀 미팅 계획

\* \*\*(1) Sprint Planning Meeting:\*\* 매주 금요일 오후 (다음 Sprint 목표와 Backlog 계획)

\* \*\*(2) Daily Scrum Meeting:\*\* 매일 지정된 시간에 30분간 진행 (진행상황 및 장애 현상 공유)

\* \*\*(3) Sprint Review \& Retrospective:\*\* Sprint 종료 시점 (리뷰 및 회고 진행)

\* \*\*(4) 비정기 회의:\*\* 프로젝트 진행에 문제가 발생하여 회의가 필요한 경우 진행



\### 4.2 변경사항 관리 방법

\* 기능 변경을 위해 팀원 전체가 권한을 가져야 하며 합의가 필요하다.

\* 변경 사항이 발생되는 경우 전-후에 대해 반드시 기록물로 남긴다.



\### 4.3 산출물 관리 방법

\* \*\*(1) 소스 코드 형상 관리 규칙 (필수 준수):\*\*

&#x20; \* 중앙 원격 저장소의 기준 브랜치는 \*\*`master`\*\*를 사용한다.

&#x20; \* 잦은 Git 충돌(Conflict)을 방지하기 위해 \*\*공용 파일(설정 파일 등)은 절대 임의로 수정하지 않는다.\*\*

&#x20; \* 각 팀원은 반드시 본인이 할당받은 UI 페이지 파일 및 독립된 모듈 영역에서만 작업하고 병합(Merge)한다.

\* \*\*(2) 산출물 명명(Naming) 방법:\*\*

&#x20; \* 문서: `\[팀명]문서명\_YYMMDD\_Doc-해당 문서의 버전` (ex. \[미니드라이브]프로젝트관리계획서\_260425\_Doc-001)

\* \*\*(3) 산출물 저장 방법:\*\*

&#x20; \* 소프트웨어/코드: GitHub 원격 저장소 (`master` 브랜치)

&#x20; \* 문서 및 기타 파일: 팀원 공유 클라우드
